# Мини веб-служба: каталог товаров

## Описание
Сервис хранит товары в памяти процесса и предоставляет:
- GET /api/items — список товаров с фильтрацией и сортировкой
- GET /api/items/{id} — товар по id или согласованная ошибка
- POST /api/items — создание товара с валидацией

Конвейер обработки запросов: request id → обработка ошибок → лимит размера → замер времени.

## Запуск
Требования: .NET 8 SDK.

```bash
dotnet run
``` 
## URL
- https://localhost:54254
- http://localhost:54255

## Проверки логики предметной области
Проверяем:

1. name не пустой
2. price не отрицательный
3. name не длиннее 150 символов
4. name уникальный (без учета регистра)

Ожидаем: 400 и единый формат ошибки с _requestId._

## Проверки взаимодействия с веб-службой
Сценарий 1: создать → получить

1. POST /api/items
2. GET /api/items/{id}

Сценарий 2: запросить несуществующий id

- GET /api/items/{id} → 404 и согласованный JSON

## Проверка логов
Для каждого запроса в логе есть:

- requestId
- timeMs
- status

Проверка: взять requestId из ответа и найти его в журнале.

## Ручной прогон (PowerShell)

Создание:
```
Invoke-RestMethod https://localhost:54254/api/items -Method Post -ContentType "application/json" -Body '{"name":"Notebook","price":1200}'
```

Список:
```
Invoke-RestMethod "https://localhost:54254/api/items"
```

Фильтрация + сортировка:
```
Invoke-RestMethod "https://localhost:54254/api/items?name=note&minPrice=100&maxPrice=2000&sort=price&order=desc"
```

Несуществующий id:

```
Invoke-RestMethod "https://localhost:54254/api/items/00000000-0000-0000-0000-000000000000"
```

> Примечание: если HTTPS не доверен, используйте http://localhost:54255 или выполните:
> ```
> `dotnet dev-certs https --trust`
> ```

## Экспериментальная часть
Независимая переменная: размер тела запроса (в пределах лимита 20 KB).  
Зависимая переменная: `timeMs` из логов.

Измерения (пример):
- Тело ~1 KB → `timeMs=3`
- Тело ~15 KB → `timeMs=9`

Вывод: с ростом размера тела растет время обработки, но остается в пределах десятков миллисекунд. Лимит 20 KB снижает риск перегрузки.

## Архитектурное пояснение
Маршруты отвечают только за бизнес-операции (CRUD).
Общие аспекты вынесены в middleware:

* request id — связывает ответ с логом
* обработка ошибок — единый формат ошибок
* лимит размера — защита на входе
* тайминг — измерение производительности

Такой конвейер — это каркас, потому что общие правила применяются ко всем запросам автоматически.

## Риски и эксплуатационные аспекты
* Ограничение Content-Length и чтение тела защищают от больших запросов.
* Валидация входных данных предотвращает некорректные значения и дубликаты.
* Логирование с requestId облегчает диагностику.

## Контрольные вопросы (кратко)
* Независимая переменная — размер тела; зависимая — timeMs из логов.
* Угрозы валидности: локальная машина и малое число запусков; уменьшал повторяемостью.
* В ядре каркаса — middleware; в приложении — маршруты и модели.
* Антипаттерны: отсутствие единого формата ошибок, разрозненная валидация.
* При росте в 10 раз — нужно БД, кэширование и пересмотр логирования.
